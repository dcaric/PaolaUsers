# Day 3 Impacts (DHMZ Weather + JWT Foundation)

## Goal
Add a simple Day 3 foundation in a separate package (`weather`) without overengineering:
- Fetch current temperatures from DHMZ XML feed.
- Support partial city-name search and return all matches.
- Add basic JWT access/refresh token issuing and validation for role handling.

## Package Added
- `src/main/java/com/paola/paolarestapi/weather`
  - `WeatherResource` (REST endpoints for weather + auth)
  - `service/DhmzWeatherService` (DHMZ XML fetch + parse + filter)
  - `service/JwtTokenService` (minimal HS256 JWT create/validate/refresh)
  - `model/WeatherTemperature`
  - `dto/*` for auth requests/responses

## Endpoints Added
Base path: `/api/weather`

1. `GET /temperature?city=zag`
- Reads `https://vrijeme.hr/hrvatska_n.xml`
- Case-insensitive partial match on city name
- Returns all matching cities and temperatures

2. `POST /auth/login`
- Request JSON:
```json
{"username":"paola","role":"read-only"}
```
- Response: access token + refresh token

3. `POST /auth/refresh`
- Request JSON:
```json
{"refreshToken":"<token>"}
```
- Response: new token pair

4. `GET /auth/validate?token=<jwt>`
- Returns whether token is valid and basic claims

## Notes
- Role normalization is intentionally simple:
  - `full-access` or `full` -> `full-access`
  - everything else -> `read-only`
- JWT secret can be configured via:
  - env var `JWT_SECRET`
  - JVM prop `-Djwt.secret=...`
- If secret is not provided, a local fallback secret is used (good for demo/dev only).

## JWT Clarification (Important)
- DHMZ does **not** send JWT tokens.
- DHMZ endpoint (`https://vrijeme.hr/hrvatska_n.xml`) provides only weather XML data.
- JWT tokens in this project are generated by our backend (`JwtTokenService`) for Day 3 security foundation work.

## Why We Use JWT
- To support role-based access logic required by the assignment (`read-only` vs `full-access`).
- To let client applications (desktop/web) carry identity + role in each request without server session state.
- To keep auth flow simple for interoperability testing across endpoints and protocols.

## How JWT Is Used Here
1. Client calls `POST /api/weather/auth/login` with username and role.
2. Backend creates and returns:
   - short-lived access token
   - long-lived refresh token
3. Client stores tokens on its side and sends access token for protected calls (future Day 4/5 full RBAC flow).
4. Client can call `GET /api/weather/auth/validate` to check token status.
5. Client can call `POST /api/weather/auth/refresh` to get a new token pair from refresh token.

## Where Tokens Are Saved
- Current implementation is stateless: backend does not persist tokens to DB/files.
- Tokens are only:
  - issued by backend in response body
  - validated by signature/claims
  - stored client-side (for example app memory or secure local storage).

## How JWT Improves Security
- Integrity: token signature (HS256) helps detect tampering.
- Expiration control: `exp` claim limits token lifetime.
- Role claim: `role` claim carries authorization intent (`read-only` / `full-access`).
- No server session table needed: less session management complexity for this stage.

Important limitation for this student project phase:
- Tokens are signed and validated, but transport security still depends on HTTPS at deployment level.
- On plain HTTP (localhost demo), traffic is not encrypted in transit.
